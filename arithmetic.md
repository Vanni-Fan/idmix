# XID v1.1 规范：自描述整数序列短标识符编码

## 1. 概述
XID 是一种紧凑、自描述的二进制到文本编码方案，专为 **只包含简单整数**（小值居多，负数偶尔出现且一般 ≥ -10）的数据序列设计。它将多个带类型的整数压缩为极短的字节序列，再通过**自定义字符表**转换为字符串，可作为 access_key、短 ID、令牌等使用。

**核心特性**：
- **类型自描述**：每个整数携带原始类型（uint8/int64 等），解码时不依赖外部 schema。
- **极致压缩**：0~15 的正数、-1~-16 的负数仅占 **1 字节**；其他整数去除冗余符号字节后按最小宽度存储。
- **32 态多态**：同一组数据可生成 32 种不同字符串，方便防猜、过滤敏感词。
- **轻量自校验**：内嵌 2-bit 校验，可即时阻挡 75% 的随机篡改或传输错误，不增加额外字节。
- **私密进制编码**：最终输出并非标准 Base64，而是采用**自定义字符集（64 字符）** 进行 Base64 无填充编码，字符顺序可作为密钥。
- **高性能**：纯二进制位操作及异或混淆，编解码极快，无需 AES 等重量级加密。

**适用场景**：
- 需在 URL、Cookie、日志中携带多个整数信息的 access_key。
- 类似 Sqids 的短 ID 生成，但要求携带更多结构化信息。
- 对防猜有一定要求，但不需要强加密的内部系统。

**限制**：
- 单次编码最多支持 **511 个**数据对象（可通过配置调整）。
- 数值范围理论上支持 uint64/int64 全范围，但推荐用于中小整数，过大的值会降低压缩率。

## 2. 二进制格式

XID 的二进制块由 **整体头（2 字节）** + **数据对象序列** 组成。

### 2.1 整体头（16 位，小端序）

将两个字节视为一个 16 位无符号整数 `header = data[0] | (data[1] << 8)`，其位分配如下（默认配置）：

| 位域        | 宽度 | 含义                |
|-------------|------|---------------------|
| bit[1:0]    | 2    | `check`  校验位      |
| bit[10:2]   | 9    | `count`  数据对象个数（0~511） |
| bit[15:11]  | 5    | `variant_id`  变体 ID（0~31） |

- `check`：初始为 0，编码时填入全局异或校验和的低 2 位。
- `count`：后续数据对象的数量。
- `variant_id`：控制多态性，32 种变体。

**字节序**：小端，即低地址存放低 8 位（`byte0 = header & 0xFF`），高地址存放高 8 位（`byte1 = (header >> 8) & 0xFF`）。

### 2.2 数据对象

每个数据对象编码一个整数及其原始类型，由 **1 字节对象头** + **可选数据负载** 构成。对象头根据最高位分为两种模式。

#### 模式 A：内嵌模式（bit7 = 0）
对象总长度 = **1 字节**。  
对象头位布局：
```
bit7   = 0
bit6   = sign       (0: 原始无符号, 1: 原始有符号)
bit5-4 = width_bits (原始位宽: 00=8位, 01=16位, 10=32位, 11=64位)
bit3-0 = value      (内嵌值 v, 0~15)
```
- 若 `sign = 0`：实际整数 = `v` （0 ~ 15）
- 若 `sign = 1`：实际整数 = `-(v + 1)` （-1 ~ -16）

内嵌模式适用于：
- **无符号类型**（uint8/16/32/64）且值在 0~15；
- **有符号类型**（int8/16/32/64）且值在 -1~-16。

有符号类型的非负整数（含 0~15）须走扩展模式。原始位宽须为 8/16/32/64 之一（由 `width_bits` 指示）。

#### 模式 B：扩展模式（bit7 = 1）
对象总长度 = 1 + **存储宽度字节数**。  
对象头位布局：
```
bit7   = 1
bit6   = 保留 (必须为 0)
bit5-4 = sw         (存储宽度: 00=1字节, 01=2字节, 10=4字节, 11=8字节)
bit3-0 = otype      (原始类型索引, 0~7)
```
后紧跟 `sw` 字节的数据负载，按**小端序**存储去除冗余后的补码（见下文）。

**原始类型索引 `otype` 映射**：

| otype | 原始类型 |
|-------|----------|
| 0     | uint8    |
| 1     | uint16   |
| 2     | uint32   |
| 3     | uint64   |
| 4     | int8     |
| 5     | int16    |
| 6     | int32    |
| 7     | int64    |

**存储宽度 `sw` 对应字节数**：

| sw   | 字节数 |
|------|--------|
| 0 (00) | 1   |
| 1 (01) | 2   |
| 2 (10) | 4   |
| 3 (11) | 8   |

**数据负载生成规则（最小补码表示）**：
- 对原始类型为无符号（otype 0~3）的数值 `V`（V ≥ 0）：  
  取 `V` 的最低有效字节，要求最高非零字节的 bit7 必须为 0。若 V=0，可用 1 字节 0x00。不允许出现最高字节 bit7=1 的情况（例如 uint16 的 128 必须用 2 字节 0x80,0x00，不可用 1 字节 0x80）。
- 对原始类型为有符号（otype 4~7）的数值 `V`：  
  取 `V` 的二进制补码（以目标宽度计算）的最低有效字节，要求被截去的高位字节必须全为符号位的扩展（正数全 0，负数全 1），且剩余部分的最高位必须与符号位一致。  
  - 正数：剩余最高字节的 bit7 必须为 0。
  - 负数：剩余最高字节的 bit7 必须为 1。  
  例如 int64(-1) 补码为全 1，1 字节 0xFF 满足，可用 sw=1；int64(128) 补码为 0x00...0080，1 字节 0x80 的 bit7=1 会误判负数，必须用 2 字节 0x80,0x00。

编码器应自动选择满足上述条件的最小 `sw`。

## 3. 多态性与混淆

使用 `variant_id` 对数据对象部分的**所有字节**（包括每个对象的头字节和负载字节）进行可逆异或混淆，使得同一组数据在不同 `variant_id` 下产生不同的二进制序列。

- 混淆掩码计算：`mask = ((variant_id * 0x9D) + 0x37) & 0xFF`
- **编码时**：对数据对象区域的每个字节执行 `byte ^= mask`
- **解码时**：读取 `variant_id`，计算相同 `mask`，再次异或即可还原。

整体头（2 字节）**不参与混淆**，以保证解码器能直接提取 `count` 和 `variant_id`。

---

## 4. 自校验

校验覆盖**整个二进制块**（整体头 + 混淆后的数据对象）。

**编码时**：
1. 将整体头中 check 位（bit1-0）暂置为 0。
2. 拼接整体头及混淆后的数据对象所有字节，计算它们的**异或和**（XOR）。
3. 取异或和的低 2 位，写入整体头的 check 位。

**解码时**：
1. 保存接收到的 `check` 值。
2. 将整体头中的 check 位清零（其他位不变），得到临时二进制块。
3. 计算临时二进制块全部字节的异或和，取低 2 位 `calc_check`。
4. 若 `calc_check != check`，则判定数据无效，拒绝解码。

此 2 位校验可阻挡 75% 的随机篡改，且不增加额外字节。

## 5. 文本编码（自定义 Base64）

最终字符串采用 **自定义字符集的 Base64 编码，无填充**。

- 字符表由用户提供，长度为 64，可任意排列。顺序不公开时可作为额外防护。
- 推荐移除易混淆字符（如 `0OIl`）后随机打乱。
- 编码方法：每 3 字节转换为 4 个字符，最后若不足 3 字节：
  - 剩余 1 字节：取该字节高 6 位、低 2 位（左移 4 位）组成两个字符，无填充。
  - 剩余 2 字节：取前 12 位组成两个字符，后 6 位组成第三个字符，无填充。
- 解码时，根据字符串长度反推原始二进制长度，不允许出现非法字符或长度错误。

**示例**：若标准 Base64 字符表为 `A-Za-z0-9+/`，自定义表可任意替换，编码算法相同。

## 6. 示例

编码三个数：`uint16(5)`, `int64(-1)`, `uint32(40)`，`variant_id`=0（掩码 0x37）。

1. **数据对象生成**：
   - `uint16(5)`：内嵌，sign=0, width_bits=01, v=5 → 对象头 = 0x15。
   - `int64(-1)`：内嵌，sign=1, width_bits=11, v=0 → 对象头 = 0x70。
   - `uint32(40)`：不能内嵌，最小补码：40 需要 1 字节 0x28（bit7=0），sw=0, otype=2 → 对象头 = 0x82，数据 0x28。
   - 数据对象序列：`15 70 82 28`。

2. **混淆**：mask=0x37，异或后：`22 47 B5 1F`。

3. **整体头**：
   - count=3, variant_id=0，header = (0<<11) | (3<<2) = 0x000C。
   - 字节：`0C 00`。
   - 加入数据对象后全块：`0C 00 22 47 B5 1F`。
   - 计算异或和：0x0C^0x00^0x22^0x47^0xB5^0x1F = 0xC3 → check=3。
   - 最终 header = 0x000F，字节：`0F 00`。
   - 最终二进制块：`0F 00 22 47 B5 1F`（6 字节）。

4. **Base64 编码**（假设标准字母表）：`DwAiR7Uf`（8 字符）。使用自定义字母表可映射为其他 8 字符。

**解码验证**：
- 解码 Base64 → 字节 `0F 00 22 47 B5 1F`
- header=0x000F → check=3, count=3, variant_id=0
- 清零 check 后异或和=0xC3 低2位=3，校验通过。
- 解混淆：mask=0x37，异或得 `15 70 82 28`
- 解析：
  - 0x15 → mode0, 无符号 uint16(5)
  - 0x70 → mode0, 有符号 int64(-1)
  - 0x82 → mode1, otype=2 (uint32), sw=0 (1字节), 数据 0x28 → 40
- 成功还原。

## 7. 配置与扩展
- 整体头位域分配可通过编译时宏或配置文件调整，例如：
  - 若不需要 32 态，可将 `variant_id` 降为 3 位，`count` 增至 11 位（最大 2047 对象）。
  - 保留位始终为 0，可作未来扩展。
- 自定义字符表建议在部署时随机生成，保证各服务实例的编码不互通。
- 内嵌模式的值域（0~15 和 -1~-16）已覆盖绝大多数小整数，若数据分布变化，可考虑调整内嵌位数（需同步修改规范）。
